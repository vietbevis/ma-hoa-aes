01 Giới thiệu AES

AES (Advanced Encryption Standard) là thuật toán mã hóa khối đối xứng được NIST công bố năm 2001 theo tiêu chuẩn FIPS PUB 197, thay thế cho DES và 3DES đã lỗi thời.

AES dựa trên thuật toán Rijndael do Joan Daemen và Vincent Rijmen (Bỉ) thiết kế, được chọn sau cuộc thi quốc tế kéo dài 5 năm (1997–2001) với 15 ứng viên.

Phiên bản Kích thước khóa Số vòng (Nr) Kích thước khối Round keys
AES-128 128 bit (16 byte) 10 128 bit 11 × 128 bit
AES-192 192 bit (24 byte) 12 128 bit 13 × 128 bit
AES-256 256 bit (32 byte) 14 128 bit 15 × 128 bit
02 Trường Galois GF(2⁸)

AES thực hiện tất cả các phép tính trên trường hữu hạn GF(2⁸) — 256 phần tử. Mỗi phần tử là một byte biểu diễn như đa thức bậc ≤ 7 với hệ số trong GF(2). Phép cộng là XOR (⊕); phép nhân là nhân đa thức mod đa thức bất khả quy. 

// Đa thức bất khả quy của AES:
// p(x) = x⁸ + x⁴ + x³ + x + 1  =  0x11B

/**
 * Nhân hai byte trong GF(2⁸) — phương pháp "shift and reduce"
 * @param {number} a - byte đầu tiên (0x00–0xFF)
 * @param {number} b - byte thứ hai
 * @returns {number} tích a × b mod p(x)
 */
function gmul(a, b) {
  let p = 0;
  for (let i = 0; i < 8; i++) {
    if (b & 1) p ^= a;              // cộng a nếu bit thấp nhất của b = 1
    const hiBit = a & 0x80;
    a = (a << 1) & 0xFF;            // nhân a với x (shift left, giữ 8 bit)
    if (hiBit) a ^= 0x1B;           // khử bậc 8: 0x11B → 0x1B (bỏ bit x⁸)
    b >>= 1;                         // dịch b phải 1 bit
  }
  return p;
}

// Ví dụ kiểm tra:
// gmul(0x57, 0x83) === 0xC1  ✓  (NIST FIPS 197, Section 4.2.1)
// gmul(0x02, 0x80) === 0x1B  ✓  (overflow → XOR 0x1B)

04 Key Expansion — Mở rộng khóa

Từ khóa gốc Nk words, sinh ra (Nr+1)×4 words để dùng cho từng vòng. AES-128: Nk=4, Nr=10, sinh 44 words. Rcon đảm bảo mỗi round key luôn duy nhất. 

/**
 * Key Expansion — FIPS 197 Section 5.2
 * Hỗ trợ AES-128 (Nk=4), AES-192 (Nk=6), AES-256 (Nk=8)
 */
function keyExpansion(keyBytes) {
  const Nk = keyBytes.length / 4;   // số words trong khóa gốc
  const Nr = Nk + 6;                // số vòng: 10 | 12 | 14
  const w  = [];

  // Sao chép khóa gốc vào w[0 .. Nk-1]
  for (let i = 0; i < Nk; i++) {
    w[i] = keyBytes.slice(i * 4, i * 4 + 4);
  }

  for (let i = Nk; i < (Nr + 1) * 4; i++) {
    let temp = w[i - 1].slice();

    if (i % Nk === 0) {
      // RotWord: [a, b, c, d] → [b, c, d, a]  (dịch trái 1 byte)
      temp = [temp[1], temp[2], temp[3], temp[0]];
      // SubWord: áp dụng S-Box cho từng byte
      temp = temp.map(b => SBOX[b]);
      // XOR với round constant Rcon[i/Nk]
      // Rcon = [-, 0x01, 0x02, 0x04, 0x08, 0x10, 0x20, 0x40, 0x80, 0x1B, 0x36]
      temp[0] ^= RCON[i / Nk];

    } else if (Nk > 6 && i % Nk === 4) {
      // Chỉ AES-256: thêm một lần SubWord khi i mod 8 = 4
      temp = temp.map(b => SBOX[b]);
    }

    w[i] = w[i - Nk].map((b, j) => b ^ temp[j]);
  }

  return { w, Nr };
}
05 SubBytes — Thay thế phi tuyến & S-Box

SubBytes thay thế mỗi byte bằng giá trị trong bảng S-Box, được xây dựng qua hai bước:

  1. Tính nghịch đảo nhân trong GF(2⁸): b = a⁻¹ mod p(x) (0x00 → 0x00)
  2. Áp dụng biến đổi affine: b'ᵢ = bᵢ ⊕ b(i+4)%8 ⊕ … ⊕ cᵢ, với hằng số c = 0x63 
// SubBytes — thay thế từng byte qua S-Box
function subBytes(state) {
  return state.map(row => row.map(byte => SBOX[byte]));
}

// InvSubBytes — dùng khi giải mã (Inverse S-Box)
function invSubBytes(state) {
  return state.map(row => row.map(byte => INV_SBOX[byte]));
}

// Một số giá trị S-Box đáng chú ý:
// SBOX[0x00] = 0x63   (do biến đổi affine với c = 0x63)
// SBOX[0x01] = 0x7C
// SBOX[0x53] = 0xED
// SBOX[0xFF] = 0x16

// Test vectors từ NIST FIPS 197 Appendix B:
// SBOX[0x19] === 0xD4  ✓
// SBOX[0xA0] === 0xE2  ✓
// SBOX[0x9A] === 0xB8  ✓
S-BOX đầy đủ 16×16 — hàng = nibble cao, cột = nibble thấp của byte đầu vào
06 ShiftRows — Dịch hàng

Mỗi hàng được dịch vòng trái (cyclic left shift) theo chỉ số hàng, tạo ra diffusion theo chiều ngang giữa các cột để chuẩn bị cho MixColumns. 

// ShiftRows — mã hóa (cyclic left shift)
function shiftRows(state) {
  return [
    [ state[0][0], state[0][1], state[0][2], state[0][3] ], // hàng 0: shift 0 (giữ nguyên)
    [ state[1][1], state[1][2], state[1][3], state[1][0] ], // hàng 1: shift 1
    [ state[2][2], state[2][3], state[2][0], state[2][1] ], // hàng 2: shift 2
    [ state[3][3], state[3][0], state[3][1], state[3][2] ], // hàng 3: shift 3
  ];
}

// InvShiftRows — giải mã (cyclic right shift = left shift bởi 4 - n)
function invShiftRows(state) {
  return [
    [ state[0][0], state[0][1], state[0][2], state[0][3] ], // shift 0
    [ state[1][3], state[1][0], state[1][1], state[1][2] ], // right 1 = left 3
    [ state[2][2], state[2][3], state[2][0], state[2][1] ], // right 2 = left 2
    [ state[3][1], state[3][2], state[3][3], state[3][0] ], // right 3 = left 1
  ];
}

// Công thức tổng quát:
// ShiftRows:    state'[r][c] = state[r][(c + r) % 4]
// InvShiftRows: state'[r][c] = state[r][(c - r + 4) % 4]
07 MixColumns — Trộn cột

Mỗi cột được nhân với ma trận MDS cố định trong GF(2⁸). Sau 2 vòng ShiftRows + MixColumns, mỗi bit output phụ thuộc tất cả 128 bit input (Wide Trail Strategy). 

// MixColumns — nhân mỗi cột với ma trận MDS trong GF(2⁸)
//
// Ma trận mã hóa:          Ma trận giải mã (nghịch đảo):
//  ┌ 2  3  1  1 ┐           ┌ 14  11  13   9 ┐
//  │ 1  2  3  1 │           │  9  14  11  13 │
//  │ 1  1  2  3 │           │ 13   9  14  11 │
//  └ 3  1  1  2 ┘           └ 11  13   9  14 ┘
//                              (14=0x0E, 11=0x0B, 13=0x0D, 9=0x09)

function mixColumn(col) {
  const [s0, s1, s2, s3] = col;
  return [
    gmul(2, s0) ^ gmul(3, s1) ^ s2          ^ s3,
    s0          ^ gmul(2, s1) ^ gmul(3, s2) ^ s3,
    s0          ^ s1          ^ gmul(2, s2) ^ gmul(3, s3),
    gmul(3, s0) ^ s1          ^ s2          ^ gmul(2, s3),
  ];
}

function mixColumns(state) {
  const out = Array.from({ length: 4 }, () => new Array(4));
  for (let c = 0; c < 4; c++) {
    const mixed = mixColumn([state[0][c], state[1][c], state[2][c], state[3][c]]);
    for (let r = 0; r < 4; r++) out[r][c] = mixed[r];
  }
  return out;
}
08 AddRoundKey — XOR với Round Key

Bước duy nhất phụ thuộc trực tiếp vào khóa bí mật. XOR là tự nghịch đảo — khi giải mã bước này dùng lại hàm mã hóa, không cần hàm ngược. 

// AddRoundKey — XOR state với round key tương ứng từ key schedule
function addRoundKey(state, w, round) {
  // w[round*4 + c] là word thứ c của round key tương ứng
  return state.map((row, r) =>
    row.map((byte, c) => byte ^ w[round * 4 + c][r])
  );
}

// Tính chất tự nghịch đảo (involutory):
//   addRoundKey(addRoundKey(state, w, r), w, r) === state
// vì:  (a ⊕ k) ⊕ k  =  a   (XOR tự triệt tiêu)
//
// → Khi giải mã, sử dụng lại hàm addRoundKey hoàn toàn giống nhau,
//   chỉ thay đổi thứ tự vòng (Nr → 0 thay vì 0 → Nr)
09 PKCS#7 Padding

AES xử lý đúng 16 bytes mỗi lần. PKCS#7 (RFC 5652) bổ sung n bytes, mỗi byte có giá trị n, với n = 16 - (len % 16). Nếu len đã là bội số 16, vẫn thêm đủ 16 bytes giá trị 0x10. 

// PKCS#7 Pad — thêm n bytes có giá trị n  (n ∈ [1, 16])
function pkcs7Pad(data) {
  const n = 16 - (data.length % 16);
  return [...data, ...Array(n).fill(n)];
}

// PKCS#7 Unpad — đọc byte cuối, kiểm tra tính hợp lệ rồi cắt
function pkcs7Unpad(data) {
  const n = data[data.length - 1];
  if (n < 1 || n > 16) throw new Error('Invalid PKCS#7 padding');
  // Kiểm tra n bytes cuối đều có giá trị n
  for (let i = data.length - n; i < data.length; i++) {
    if (data[i] !== n) throw new Error('Padding bytes mismatch');
  }
  return data.slice(0, data.length - n);
}

// Ví dụ — "Hello" (5 bytes) → n = 11 → padding 0x0B:
// 48 65 6C 6C 6F  0B 0B 0B 0B 0B 0B 0B 0B 0B 0B 0B
//
// "Two One Nine Two" (16 bytes) → n = 16 → padding 0x10:
// 54 77 6F ... 54 77 6F  10 10 10 10 10 10 10 10 10 10 10 10 10 10 10 10
⚠️ Padding chỉ dùng với ECB và CBC. Các chế độ CFB, OFB, CTR hoạt động như stream cipher — XOR trực tiếp theo độ dài thực tế, không cần padding.
10 Chế độ ECB — Electronic Codebook

Chế độ đơn giản nhất: mỗi block 16 bytes mã hóa/giải mã độc lập với cùng khóa. 

Mã hóa ECB:
    Cᵢ = AES_Encrypt(Pᵢ, key)        ← mỗi block độc lập, song song được

11 Chế độ CBC — Cipher Block Chaining

Mỗi block plaintext được XOR với ciphertext của block trước rồi mới mã hóa, ẩn hoàn toàn pattern dữ liệu. Block đầu tiên XOR với IV. 

Mã hóa CBC (phải tuần tự — không song song được):
    C₀ = AES_Encrypt(P₀ ⊕ IV,    key)
    C₁ = AES_Encrypt(P₁ ⊕ C₀,   key)
    Cᵢ = AES_Encrypt(Pᵢ ⊕ Cᵢ₋₁, key)

Giải mã CBC (có thể song song vì Cᵢ₋₁ đã biết):
    P₀ = AES_Decrypt(C₀, key) ⊕ IV
    P₁ = AES_Decrypt(C₁, key) ⊕ C₀
    Pᵢ = AES_Decrypt(Cᵢ, key) ⊕ Cᵢ₋₁

Tính chất lỗi:
    Lỗi 1 ciphertext block → ảnh hưởng đúng 2 plaintext block kề nhau
⚠️ IV phải ngẫu nhiên và duy nhất cho mỗi lần mã hóa. IV không cần bí mật nhưng phải gửi kèm ciphertext.

12 Chế độ CFB — Cipher Feedback

Biến AES thành stream cipher: mã hóa shift register, XOR với plaintext; ciphertext được đưa lại làm input tiếp theo (feedback). Chỉ cần AES_Encrypt — không dùng AES_Decrypt. 

Mã hóa CFB (tuần tự — ciphertext làm feedback):
    O₀ = AES_Encrypt(IV,  key)
    C₀ = P₀ ⊕ O₀

    O₁ = AES_Encrypt(C₀, key)      ← C₀ (ciphertext) làm input tiếp theo
    C₁ = P₁ ⊕ O₁

    Oᵢ = AES_Encrypt(Cᵢ₋₁, key)
    Cᵢ = Pᵢ ⊕ Oᵢ

Giải mã CFB (có thể song song — Oᵢ chỉ phụ thuộc ciphertext):
    Oᵢ = AES_Encrypt(Cᵢ₋₁, key)  ← vẫn dùng AES_Encrypt, không phải Decrypt
    Pᵢ = Cᵢ ⊕ Oᵢ

Tính chất:
    • Không cần padding (stream cipher)
    • Lỗi 1 bit ciphertext → ảnh hưởng 2 block plaintext liên tiếp

13 Chế độ OFB — Output Feedback

Keystream hoàn toàn độc lập với dữ liệu — feedback là output của AES (không phải ciphertext). Keystream có thể tính trước khi chưa có plaintext. 

Tạo keystream OFB (chỉ phụ thuộc key và IV — tính trước được):
    K₀ = AES_Encrypt(IV,  key)
    K₁ = AES_Encrypt(K₀, key)      ← output của AES làm feedback (khác CFB!)
    K₂ = AES_Encrypt(K₁, key)
    Kᵢ = AES_Encrypt(Kᵢ₋₁, key)

Mã hóa và giải mã hoàn toàn đối xứng (chỉ là XOR):
    Cᵢ = Pᵢ ⊕ Kᵢ
    Pᵢ = Cᵢ ⊕ Kᵢ

Tính chất:
    • Lỗi 1 bit ciphertext → chỉ ảnh hưởng đúng 1 bit plaintext
    • Keystream phải tính tuần tự (không song song)
    ⚠ Tái sử dụng (IV, key): biết P₀ → tính được P₀' của thông điệp khác!
⚠️ Tuyệt đối không tái sử dụng (IV, key) trong OFB. Keystream không thay đổi → two-time pad attack.
14 Chế độ CTR — Counter Mode

Mã hóa một counter tăng dần và XOR với dữ liệu. Song song hóa hoàn toàn, hỗ trợ random access — lựa chọn hàng đầu trong TLS 1.3, QUIC, SSH. 

// CTR — Counter Mode
// Counter block layout: [ nonce (8 bytes) | counter (8 bytes, big-endian) ]

/**
 * Mã hóa / giải mã CTR (hoàn toàn đối xứng)
 * @param {number[]} data  - plaintext hoặc ciphertext
 * @param {number[]} key   - khóa AES (16|24|32 bytes)
 * @param {number[]} nonce - 8 bytes ngẫu nhiên, duy nhất cho mỗi message
 */
function ctrCrypt(data, key, nonce) {
  const counterBlock = [...nonce, 0, 0, 0, 0, 0, 0, 0, 0]; // 16 bytes

  return data.map((byte, i) => {
    if (i % 16 === 0 && i > 0) {
      // Tăng counter (big-endian, 8 bytes cuối)
      for (let j = 15; j >= 8; j--) {
        if (++counterBlock[j] !== 0) break; // carry
      }
    }
    // Lấy byte keystream tương ứng (chỉ mã hóa block khi cần)
    const blockIdx  = Math.floor(i / 16);
    const byteInBlk = i % 16;
    const ksBlock   = AES_Encrypt(counterBlock, key);
    return byte ^ ksBlock[byteInBlk];
  });
}

// Random access — giải mã block thứ N không cần giải các block trước:
//   counterBlock[8..15] = N
//   keystream_N = AES_Encrypt(counterBlock, key)
//   plaintext_N = ciphertext_N XOR keystream_N
ℹ️ CTR kết hợp với Galois Message Authentication → AES-GCM: mã hóa xác thực (AEAD) chuẩn của TLS 1.3, đảm bảo cả bảo mật lẫn toàn vẹn dữ liệu.

16 Độ an toàn và ứng dụng thực tế

AES-128 có 2¹²⁸ ≈ 3.4×10³⁸ khóa — không thể brute-force. Cuộc tấn công tốt nhất đã biết là biclique cryptanalysis (2011), chỉ đạt ~2¹²⁶·¹ — hoàn toàn không khả thi trong thực tế.
